  一、为什么 stream_generate 有效、stream_generate_step 几乎无效
  
  这两条路径对训练分布的契合度差太多。

  训练时模型实际看到的是什么

  forward() → _forward_single_window()：

  1. 整段 feature 是完整的 (B, T_token, C) clean latent，随机选一个 time_step
  2. 整段 traj 通过 _build_traj_emb 编码（一次性看到 0..T_token 的全部轨迹）
  3. ControlNet 对 noisy_input[:, :end_index] 做条件，traj_emb 对 batch
  内每个样本只编一次
  4. traj 的 anchor 从来没有被减掉——encode_traj_batch 直接把 traj xyz 经 heading
   编码，绝对坐标进 ControlNet
  5. self-forcing K 步 rollout 时也是用整段 traj_emb（self_forcing.py:309-311
  一次构造，K 步共享）

  stream_generate 看到的是什么（metrics 用的）

  diffusion_forcing_wan.py:1107-1109
  gen_seq_len = seq_len + self.chunk_size
  traj_emb = self._build_traj_emb(x, gen_seq_len, device)
  traj_seq_lens = self._get_traj_seq_lens(x, gen_seq_len, device)

  整段 traj 一次编完，跟训练几乎一致。✓ 这就是它效果好的原因。

  stream_generate_step + web_demo 看到的是什么

  stream_generate_step:1330-1340：
  traj_emb = self._traj_buf.build_traj_emb(end_index, self.seq_len, device)

  build_traj_emb 内部走 _build_from_xyz，做了几件训练时从来不做的事：

  1. anchor-subtract：traj_slice = traj_slice - 
  anchor.unsqueeze(1)（traj_stream_buffer.py:303）
  2. token-level xyz → frame-level lerp 重建 （_expand_tokens_to_causal_frames）
  3. 再 root_to_traj_feats 算 heading

  训练时是 frame-level traj 直接做 heading，从来不做 anchor 减法、不做
  token→frame lerp。

  模型从未在"anchor 已减、xyz 经过 lerp 重建"的 traj feature 分布下训过。
  这是分布外推理。

  更糟的是：web_demo 传入的是 traj_horizon_tokens=20 个未来 
  token（model_manager.py:447），但 stream_generate_step 滑动窗口是
  seq_len=2*history_length（init_generated:1290）。build_traj_emb(end_index, 
  self.seq_len) 取 [end_index-seq_len, end_index) 这段，末端 traj 几乎全是 
  0（buffer 没填）。即使 get_traj_valid_lens 修过 attention mask 让那部分不被
  attend，有效 traj 长度也只是 min(traj_horizon_tokens, …)，远短于训练时的全段
  traj。

  第二个问题：history vs full-sequence

  训练 forward()：模型把整段 feature[:, :end_index] 当 context，左 clean、右
  noisy，context 长度从 chunk_size 到 max_length 都覆盖。

  stream_generate_step：context 永远是 [-seq_len:] 这个 2*history_length
  大小的滑动窗口（1320-1322）。当 commit_index < seq_len 时窗口前缘是 padding
  noise，跟训练分布相似；当 commit_index >= seq_len
  时窗口左边界开始裁掉真正的早期 token——这种"已经走过很长时间的 sliding 
  context"训练时完全不存在。

  web_demo 默认 history_length=20，所以 seq_len=40。运行几秒后 commit_index
  就会绕回（1414-1435 的 roll），buffer 里只有最近 40 token 的内容。但 traj 在 
  buffer 里也只是相对的、anchor-subtracted 的近 40 
  token——绝对世界坐标信息完全丢失。

● 你的观察非常准确。run_eval.py 走的是 generate()（offline 全 traj），那是训练分布；任何离开这条路径的都是 OOD。要让流式真正工作，需要把训练分布主动往流式推理对齐，而不是反过来。

  ---
  一、文本能热切换、轨迹不能，根因是什么
  
  文本热切换之所以无缝，是因为：

  1. 文本是高层语义，不是空间硬约束——上一句"walk"和下一句"run"在 latent 空间里距离近，模型用一两个 token 就能过渡
  2. 训练时 text_condition_list 本来就是 per-token 的——multi-text 样本（isinstance(text_list[0], list)）让模型见过 per-token 切换的分布
  3. 文本不携带"当前位置"信息，所以切换后没有"几何不连续"问题

  轨迹热切换难，因为：

  1. 轨迹是空间硬约束——新轨迹的第一个点必须从当前 root 位置开始，否则就是"瞬移"
  2. 当前训练用的是绝对世界坐标（traj_features = [x, z, cos, sin]，前两维是绝对 xz）——切换 traj 意味着告诉模型"忽略你刚才走到的位置，从我给的新位置开始"，模型没学过这件事
  3. 训练时整段 traj 一次给完，模型从未见过"前一段是 traj A、后一段切换到 traj B"的拼接分布

  → 想让轨迹像文本一样热切换，必须把这三点逐一解掉。

  ---
  二、三个核心设计决策

  决策 1：轨迹表示要从"绝对"改成"相对当前根"

  这是最重要的一条。

  当前 traj_features = [x_abs, z_abs, cos ψ, sin ψ]，绝对坐标。流式推理时 web_demo 已经做了 anchor-subtract，把绝对坐标变成相对——但训练没见过这个分布，所以坑就在这。

  更好的设计：直接训练在"相对当前 root 的位移 + 朝向"上。

  具体选项：
  - A1（最简单）：traj_features = [Δx, Δz, cos ψ, sin ψ]，其中 Δ = position − anchor_position，anchor 是当前 commit_index 对应的 root 位置
  - A2（更强）：traj_features = [Δx_local, Δz_local, cos ψ_local, sin ψ_local]，相对当前 root 还要旋转到 local frame（消除朝向偏置）

  A2 跟 motion_tracking 数据集的 local_body_pos 语义一致，最贴近 G1 reference 的语义。但 A1 已经够用，A2 是锦上添花。

  带来的副作用：训练时怎么定义 anchor？
  - 训练时每个 sample 的 anchor 可以是 frame 0（等价于"clip 起点对齐"）
  - 也可以在数据增强里随机选一个 anchor token，模拟"流式中已经走了一段"
  - self-forcing rollout 里的 anchor 跟 rollout 起点对齐

  这样训练分布就天然包含了"未来轨迹是相对当前位置的"语义，热切换时直接换轨迹起点 = 当前 root 就成立。

  ---
  决策 2：轨迹必须 horizon-limited 训练
  
  当前训练时模型看到的是整段 traj（200 帧），但 web_demo 只给 20 token（80 帧）的未来。

  模型从未学过"只看未来 20 token"的条件分布。 这就跟问"只读到第一段的 essay 怎么续写"差不多——OOD。

  具体改法：
  - 训练时在 encode_traj_batch 里随机截断 traj_features，保留前 H 个 token（H ∈ [5, 30] 随机），其余清零并 token_mask=0
  - 这样模型同时学到：H=5 时"短视未来"、H=30 时"长视未来"、H 介于之间的渐变
  - 推理时 traj_horizon_tokens=20 完全在训练分布内
  
  注意：这跟"稀疏 mask"是不同的。稀疏 mask 是"未来 100 token 但只观察其中 20 个"，horizon truncation 是"只观察未来 20 个，再之后未知"。语义不同，推理需求是后者。

  ---
  决策 3：训练里显式加入"轨迹切换"增强

  这是真正解决"热切换无缝"的关键。

  类比文本里 text_list[0] is list 的 multi-text 训练——多段文本在一个 sample 里拼接，token 边界明确。轨迹要做同样的事：

  Trajectory splice augmentation：

  1. 训练 batch 里以一定概率（比如 30%）做"轨迹拼接"：
    - 从同 batch 里挑两个样本 A、B
    - 在样本 A 的 token 序列里随机选一个切换点 t_split ∈ [chunk_size, T_A - chunk_size]
    - 把 A 前段 + B 后段拼接成"复合轨迹"传给 ControlNet
    - 但 feature（GT 动作）仍然只来自 A——模型看到的轨迹后段跟动作 GT 不一致
  2. 训练目标分两种思路：
    - 思路 X（强约束）：要求模型从 t_split 开始预测的动作能跟拼接后的 B 轨迹一致。但这需要重新组合 GT 动作，工作量大
    - 思路 Y（弱约束，推荐）：在 t_split 处给模型一个"软切换提示"——把切换点之后的 traj 当作"新规划"，但 control loss 只在 t_split 之前算（已经走过的部分）。模型自然学会"传给我未来轨迹时我跟着走、切换时不慌"

  思路 Y 实施起来简单：
  # 在 dataset 或 collate 里，30% 概率做轨迹拼接
  if random.random() < 0.3 and len(batch) >= 2:
      a, b = batch[0], batch[1]
      t_split = random.randint(chunk_size, len(a['token']) - chunk_size)
      # 把 b 的 traj 后段接到 a 上
      a['traj_features'][4*t_split:] = b['traj_features'][:len(a)-4*t_split]
      # control loss 的 traj GT 也要相应处理 —— 思路 Y 只算 t_split 之前的
      a['traj_mask'][4*t_split:] = 0  # 切换点之后不算 control loss
      
  这就跟 multi-text 训练等价：文本可以中段换、轨迹也可以中段换、训练分布覆盖了切换分布。

  ---
  三、Stream 推理路径的修复
  
  修了上面三件事，stream_generate_step 还需要顺手做几件事，让它跟训练完全对齐：

  3.1 _build_from_features 必须经过 LocalTrajEncoder（之前发现的 P0）

  这一条我前面已经说过，必须修。

  3.2 Buffer 不再做 anchor-subtract

  如果决策 1 在训练里加了 relative 表示，buffer 就不应该再减 anchor——传给 buffer 的本来就是 relative 的；如果传的是绝对的，由 web_demo._build_stream_traj_input 在送进 buffer 之前减好，buffer 内只存"训练时见过的格式"。

  3.3 滑动窗口的 anchor 跟 commit_index 对齐

  每次 commit_index 推进时，"当前 root"也在动。如果 buffer 存的是相对 anchor 的坐标，anchor 需要跟着 commit_index 一起更新：

  - commit_index=0 时 anchor=root[0]
  - commit_index=10 时 anchor=root[10]
  - 这样 buffer 里"future traj 离 anchor 的距离"始终代表"从当前 root 看去未来要走多远"

  这就是 web_demo._build_remaining_polyline 里 current_root = stream_recovery.r_pos_accum 的作用——每步重新投影 polyline 到 current root。但 stream_generate_step 内部的 buffer 没做这个滚动 anchor，只有第一帧的 anchor 是 0。这是热切换失败的另一个根因：切换
   traj 后 anchor 没更新，模型看到的 future traj 在错误的坐标系里。

  四、跟 inline metric 的关系

  你说"训练出来的模型只在 run_eval.py 中有效"——非常准确的诊断。

  run_eval.py 跑的是 generate()，全段 traj、绝对坐标、整段一次给完——这跟训练一致。

  stream_generate 跑的是流式但全段 traj——这是中间状态：滑动窗口跟训练不一致，但 traj 输入还算贴近。

  stream_generate_step 跑的是流式 + horizon-limited traj + buffer anchor 处理——这跟训练相差最远。

  我建议的验证顺序（也是难度递增）：

  ┌────────────────────────────────────────────────────────┬──────────────────────┬────────────────────────┐
  │                          路径                          │ 输入分布与训练的差距 │       应当的效果       │
  ├────────────────────────────────────────────────────────┼──────────────────────┼────────────────────────┤
  │ generate()                                             │ ≈ 0                  │ 应该最好（当前已验证） │
  ├────────────────────────────────────────────────────────┼──────────────────────┼────────────────────────┤
  │ stream_generate(全段 traj)                             │ 滑动窗口             │ 应该接近 generate      │
  ├────────────────────────────────────────────────────────┼──────────────────────┼────────────────────────┤
  │ stream_generate_step(全段 traj, GT root)               │ + 逐 token           │ 应该接近 generate      │
  ├────────────────────────────────────────────────────────┼──────────────────────┼────────────────────────┤
  │ stream_generate_step(全段 traj, 预测 root)             │ + 累积漂移           │ 略差                   │
  ├────────────────────────────────────────────────────────┼──────────────────────┼────────────────────────┤
  │ stream_generate_step(horizon=20, 预测 root)            │ + horizon 限制       │ 应该接近 demo          │
  ├────────────────────────────────────────────────────────┼──────────────────────┼────────────────────────┤
  │ stream_generate_step(horizon=20, 预测 root, swap traj) │ + 热切换             │ 真正的 demo 场景       │
  └────────────────────────────────────────────────────────┴──────────────────────┴────────────────────────┘

  加 metric 时建议这六条全部测一遍——这样能精准定位"哪一层引入了 OOD"。当前 metric 只测了最理想的几条，真正 demo 跑的最后一条没测。

  五、我的推荐方案（finetune 不重训）

  按优先级和实施难度：

  Step 1：先验证假设（不改训练，1-2 天）

  写一个完整的 stream metric，按上面表格的六条路径都测一遍。用同一个当前 ckpt 跑，看 control loss 在哪一行从"还可以"跌成"无效"。这一步能精准告诉你瓶颈在哪。

  Step 2：修 stream 路径的对齐 bug（修代码，不重训，半天）

  - _build_from_features 补 LocalTrajEncoder
  - _build_from_xyz 去掉 anchor-subtract（暂时让 stream 直接用绝对坐标，跟训练一致）
  - 这一步不需要重训，理论上能让 stream_generate_step(全段 traj, GT root) 的效果立刻接近 generate()

  Step 3：finetune 加 horizon truncation（10-30k step）

  在 encode_traj_batch 里加随机 horizon 截断，从当前 ckpt resume。
  这一步让模型学会"只看未来 H token"的条件分布。

  Step 4：finetune 加相对坐标 + swap augmentation（30-60k step）

  - traj_features 改成 relative-to-anchor，anchor 在训练里随机偏移
  - 30% 概率做 trajectory splice
  - buffer 里同步 anchor 滚动

  这一步是为热切换打基础。Step 3 完成后 stream 已经能跟随，Step 4 让它能切换。

  ---
  一句话总结

  文本能无缝切换是因为训练分布天然包含了 per-token 切换（multi-text 样本）；轨迹想做到同样的无缝，必须把训练分布主动改成"相对当前根、horizon 受限、可中段切换"——而不是寄希望于在绝对坐标上训练的模型在推理时"硬切" traj 还能不崩。

  第一步先把 stream 路径跟训练分布对齐（修 LocalTrajEncoder 跳过 + 去 anchor-subtract）；这一步不重训就能让 stream 接近 offline。然后再 finetune 学 horizon + swap 才能解锁热切换。
