● 完整复盘



  1. 现象



  旧代码训练，从 335000 resume 到 392000。训练日志里 inline eval 对 test set 样本 000021 的 ADE=0.25。但用 run_eval.py 加载同一步保存的 step_392000.ckpt，同一样本 ADE=0.95（4x 差距）。388000 等相邻 step 也有同样问题。



  2. 排除重构



  怀疑新代码重构引入了行为差异。用 sanity/ 目录（旧版代码完整备份）跑 eval_generation_metrics.py 加载同一个 ckpt，结果同样是 0.95。确认重构不是原因。



  进一步验证：新代码从 335000 resume 训练，loss 曲线和 inline eval 数值与旧代码完全一致。同一样本 000021 在旧代码和新代码的 inline eval 中表现相同（ADE=1.046 at 336000）。两条训练路径的行为一致，重构没有引入任何偏差。



  3. 定位差异来源：inline vs standalone



  同一个 checkpoint，inline eval 和 run_eval.py 结果不同。对比两条路径的完整调用链：



  - inline：训练过程中 validation 触发 → test_step → run_inline_generation_eval → average_parameters（EMA 应用到模型）→ generate() → VAE decode

  - standalone：run_eval.py → trainer.test(ckpt_path=...) → on_load_checkpoint（加载 state_dict + ema_state） → 同上 run_inline_generation_eval



  两者最终走同一个 run_inline_generation_eval 函数，EMA 应用的代码相同，seed 相同（_stable_eval_seed 保证确定性），text 和 feature 的 hash 也一致。



  4. 密集 hash 日志排查



  在关键位置加了 SHA256 hash 对比：



  ┌─────────────┬─────────────────────────────────────────────────────────────────────┐

  │    位置     │                                说明                                 │

  ├─────────────┼─────────────────────────────────────────────────────────────────────┤

  │ [SAVE]      │ on_save_checkpoint 调用时：state_dict hash、EMA hash、snapshot hash │

  ├─────────────┼─────────────────────────────────────────────────────────────────────┤

  │ [PRE-EMA]   │ inline eval 的 average_parameters 进入前：state_dict hash、EMA hash │

  ├─────────────┼─────────────────────────────────────────────────────────────────────┤

  │ ===         │ average_parameters 内部（EMA 已应用）：state_dict hash、EMA hash    │

  ├─────────────┼─────────────────────────────────────────────────────────────────────┤

  │ [POST-EMA]  │ 退出 average_parameters 后：是否恢复到 PRE-EMA 值                   │

  ├─────────────┼─────────────────────────────────────────────────────────────────────┤

  │ [EVAL-LOAD] │ run_eval.py 加载 ckpt 并应用 ema_applied_trainable 后               │

  └─────────────┴─────────────────────────────────────────────────────────────────────┘



  发现关键矛盾：



  [SAVE step=335100]     ema_hash=d01ed478    ← ckpt 保存时的 EMA

  [PRE-EMA step_335100]  ema_hash=63f73552    ← inline eval 前的 EMA ≠ SAVE



  SAVE 的 EMA 和 inline eval 用的 EMA 不同。但 [POST-EMA] 显示 sd_restored=True（EMA 应用和恢复都正常），standalone 加载的 EMA 和 SAVE 一致（序列化无损）。问题锁定：ckpt 保存的 EMA 和 inline eval 用的 EMA 在同一个 step 内不一致。



  5. 找罪魁祸首



  给 EMA.update() 加 monkey-patch 记录每次调用的 stack trace，同时在 on_save_checkpoint 也加 stack trace。



  发现 ModelCheckpoint._save_monitor_checkpoint 在 Lightning 事件循环的 on_validation_end 阶段被调用，它在模块的 on_train_batch_end（即 ema.update() 的调用时机）之前执行。



  Lightning 的执行顺序：



  step 335099 结束

    ├── on_train_batch_end → ema.update() → EMA = d01ed478

    └── validation

          └── on_validation_end

                └── ModelCheckpoint._save_monitor_checkpoint

                      └── on_save_checkpoint → EMA = d01ed478  ← 存的旧 EMA！

                                                         文件名是 step_335100.ckpt



  step 335100 开始

    ├── on_train_batch_end → ema.update() → EMA = 63f73552

    └── validation → inline eval → 用的 63f73552



  ModelCheckpoint 在 validation 结束时保存，此时 global_step 已标记为 335100（用于文件名），但 ema.update() 还没被调用（那是下一个 on_train_batch_end 的事）。于是 step_335100.ckpt 里的 EMA 实际是 335099 的。



  6. 修复与验证



  第一阶段：在 eval_runner.py 的 inline eval 中保存 EMA 快照到独立文件。standalone 加载该文件而非 ckpt 中的 EMA。335100 验证通过——standalone 的 sd_hash 等于 inline eval 的 sd_hash (7dcb8a5f)。



  第二阶段（最终方案）：去掉独立快照文件，改为在 on_validation_epoch_end 里通过 _resave_checkpoint_with_eval_ema() 重新调用 trainer.save_checkpoint() 覆盖同路径的 ckpt。此时 on_save_checkpoint 捕获到的 EMA 是 inline eval 用的那一版，直接写入

  ema_applied_trainable。run_eval.py 加载时优先使用 ema_applied_trainable，完全一致。



  7. 未完全解释的部分



  相邻步 EMA 差异量级约 1e-6，应用到生成上的影响远不足以解释旧 run 中 0.25→0.95 的 4x 差距。后续新 run 中观察到的 1.046→0.93（~12% gap）更符合一步 EMA 漂移的合理量级。旧 run 的 392000 的大 gap 可能还混入了当时排查中未复现的其他因素（dataloader text

  selection、不同 GPU 调度等）。
