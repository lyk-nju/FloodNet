# FloodNet 流式轨迹控制调试记录

本文记录最近围绕 `web_demo` / `stream_generate_step()` 做的诊断实验、结果和当前判断。

目标不是替代长期 TODO，而是保存已经验证过的事实，避免后续重复排查。

---

## 1. 背景问题

现象：

- `generate()` / `stream_generate()` 离线或整段流式生成时，轨迹跟随效果基本正常。
- `web_demo` 使用 `stream_generate_step()` 逐 token 推理时，手动 `update_traj` 后效果不稳定：
  - 有时不会按新方向转身，而像平移过去。
  - 自动 repeat 后如果轨迹没有正确平移到当前 root 附近，会出现往回走、循环摆动。
  - `clear_traj` 后也曾出现残留轨迹影响。

核心怀疑点：

- 是否 `stream_generate_step()` 本身有 bug。
- 是否 history window / horizon 不够。
- 是否 GT-root 与 pred-root 闭环差异导致误差反馈。
- 是否 update 轨迹写入正在 denoise 的窗口，造成条件突变。
- 是否 web_demo 的轨迹时间参数与诊断脚本不一致。

---

## 2. 指标定义

诊断脚本主要看 root 的 XZ 平面轨迹。

| 指标 | 含义 |
|---|---|
| `ADE` | 所有帧上 `pred_root_xz` 与目标 root XZ 的平均欧氏距离 |
| `FDE` | 最后一帧的 root XZ 距离 |
| `pre` / `ADE_pre_split` | mid-session update 生效点之前的 ADE |
| `post` / `ADE_post_split` | mid-session update 生效点之后的 ADE |
| `arc_ADE` | 按路径弧长重新参数化后的 ADE，用来弱化相位快慢影响 |
| `chamfer` | one-sided path chamfer，表示生成路径是否贴近目标路径形状 |
| `path_ratio` | `pred_path_length / gt_path_length`，看整体走得过长还是过短 |

注意：

- `post` 是时间帧对齐误差，不是只看路径形状。
- 如果模型路径形状对了但走快/走慢，`post` 可能高，但 `arc_ADE` / `chamfer` 会更低。
- 对 delayed update，`post` 从实际生效 token 对应的 frame 开始算，而不是原始用户触发 token。

## 3. 单样本 step 路径诊断

样本：

- `HumanML3D 001168`
- 文本：`a person walks forward casually with a swagger to their hips...`
- checkpoint：`FloodNet/outputs/step_460000.ckpt`
- VAE：`FloodNet/outputs/vae_1d_z4_step=300000.ckpt`

命令模板：

```bash
cd /home/yuankai/Text2Motion/FloodNet
python eval/diagnose_stream_control.py \
  --config configs/stream.yaml \
  --ckpt /home/yuankai/Text2Motion/FloodNet/outputs/step_460000.ckpt \
  --vae_ckpt /home/yuankai/Text2Motion/FloodNet/outputs/vae_1d_z4_step=300000.ckpt \
  --sample_id 001168 \
  --raw_data_dir /data1/yuankai/text2Motion/FloodDiffusion/raw_data
```

### 3.1 基础矩阵

| Mode | ADE | FDE | 结论 |
|---|---:|---:|---|
| `generate_full` | 0.1405 | 0.3066 | 离线 baseline 正常 |
| `stream_generate_full` | 0.1815 | 0.3388 | 整段流式正常 |
| `step_full_xyz_gtroot` | 0.1350 | 0.2969 | 逐 token + GT root 正常 |
| `step_full_xyz_predroot` | 0.7071 | 0.1721 | pred-root 闭环明显变差 |
| `step_h20_xyz_gtroot` | 0.0947 | 0.3653 | H=20 + GT root 正常 |
| `step_h20_xyz_predroot` | 0.7242 | 0.3698 | H=20 + pred-root 闭环仍差 |

结论：

- `stream_generate_step()` 推理主路径本身不是主要问题。
- 只要轨迹构造使用 GT root，step 路径可以达到和离线类似的 ADE。
- 真正 gap 是 pred-root 闭环：模型生成的 root 一旦偏离，后续相对轨迹 anchor 跟着变，误差会反馈到下一步轨迹条件里。

### 3.2 step-path 细分

命令：

```bash
python eval/diagnose_stream_control.py \
  --config configs/stream.yaml \
  --ckpt /home/yuankai/Text2Motion/FloodNet/outputs/step_460000.ckpt \

  --sample_id 001168 \

```

| Mode | ADE | FDE | 结论 |
|---|---:|---:|---|
| `step_full_features_gtroot` | 0.5283 | 0.2109 | feature 路径不如 xyz 路径 |
| `step_full_xyz_gtroot` | 0.0882 | 0.2833 | xyz + GT root 很好 |
| `step_no_traj` | 0.5206 | 0.6106 | 无轨迹明显变差 |
| `step_offline_decode` | 0.0748 | 0.2963 | VAE decode 本身不是瓶颈 |

结论：

- 轨迹条件确实生效。
- `xyz` 轨迹输入比绕过 `LocalTrajEncoder` 的 feature 路径更可靠。
- 问题不是 ControlNet 完全没作用，而是闭环构造和 update 策略导致轨迹条件变成 OOD。

## 4. pred-root 轨迹补偿与时间戳实验

### 4.1 token step / plan speed

结果：

| Mode | ADE | FDE | token_step |
|---|---:|---:|---:|
| `predroot_step_online_estimate_h20` | 0.6814 | 0.1171 | 0.1002 |
| `predroot_step_fixed_0.0600_h20` | 0.5452 | 1.2329 | 0.0600 |
| `predroot_step_fixed_0.0800_h20` | 0.3833 | 0.6829 | 0.0800 |
| `predroot_step_fixed_0.1000_h20` | 0.3791 | 0.0893 | 0.1000 |
| `predroot_step_fixed_0.1200_h20` | 0.3832 | 0.1687 | 0.1200 |
| `predroot_step_fixed_0.1500_h20` | 0.6286 | 0.2609 | 0.1500 |
| `predroot_step_fixed_0.2000_h20` | 0.9584 | 0.2781 | 0.2000 |

| `predroot_step_oracle_h20` | 0.3669 | 0.0828 | 0.0979 |

结论：

- 固定 `token_step` 接近 oracle 时能明显改善 pred-root 闭环。
- 过快的默认速度会放大闭环偏差。
- web_demo 不应该依赖粗糙的在线估计速度长期运行。

### 4.2 timestamped / duration waypoint

结果：

| Mode | ADE | FDE | future_step | XTE |
|---|---:|---:|---:|---:|
| `gtroot_h20` | 0.0751 | 0.2753 | nan | nan |

| `predroot_oracle_h20` | 0.3669 | 0.0828 | 0.0654 | 0.0527 |
| `timestamped_gt_h20` | 0.0735 | 0.2172 | 0.0706 | 0.0346 |

结论：

- 带真实时间语义的 `timestamped_gt_h20` 最接近 GT-root 上限。
- 只给总时长、用 `0.05s` 采样的 duration waypoint 效果明显优于原始 predroot 当前策略。
- `0.05s` 对齐 HumanML3D 20 FPS，是 web_demo 当前推荐默认。
- 该模式虽然 ADE 不如 GT-root，但视频上表现为更符合预期的匀速轨迹。

## 6. mid-session update 实验

目标：

- 模拟 web_demo 中途 `update_traj`。
- 在原路径某个 token 后，把未来路径旋转 30 度，观察模型是否能跟随转向。
- 比较立即更新、延迟 5 token、延迟 20 token 的效果。

命令：

```bash
cd /home/yuankai/Text2Motion/FloodNet
python eval/diagnose_stream_control.py \
  --config configs/stream.yaml \
  --ckpt /home/yuankai/Text2Motion/FloodNet/outputs/step_460000.ckpt \
  --vae_ckpt /home/yuankai/Text2Motion/FloodNet/outputs/vae_1d_z4_step=300000.ckpt \
  --sample_id 001168 \
  --raw_data_dir /data1/yuankai/text2Motion/FloodDiffusion/raw_data \
  --mid_session_update \
  --split_token 15 \
  --update_angle 30 \
  --render_video
```

设置：

- `requested_split_token = 15`
- `update_angle = 30 deg`
- `mid_update_delay_tokens = 0,5,20`
- `mid_update_extra_tokens = 20`
- `d0` 表示立即更新。
- `d5` 表示当前 chunk 后更新。
- `d20` 表示当前 horizon 后更新。

结果：

| Mode | ADE | pre | post | arc_ADE | chamfer |
|---|---:|---:|---:|---:|---:|
| `mid_update_time_cont_split15_rot30` | 0.0678 | 0.0173 | 0.1022 | 0.0828 | 0.0232 |
| `mid_update_arc_cont_split15_rot30` | 0.3958 | 0.4317 | 0.3714 | 0.0534 | 0.0518 |
| `mid_update_time_split_d0_split15_rot30` | 0.7236 | 0.0174 | 0.9691 | 0.5187 | 0.6063 |
| `mid_update_arc_split_d0_split15_rot30` | 0.8231 | 0.3015 | 1.0044 | 0.7986 | 0.6419 |
| `mid_update_time_split_d5_split15_rot30` | 0.5727 | 0.0290 | 0.8280 | 0.3858 | 0.4080 |
| `mid_update_arc_split_d5_split15_rot30` | 0.8022 | 0.3795 | 1.0007 | 0.9507 | 0.2659 |
| `mid_update_time_split_d20_split15_rot30` | 0.4452 | 0.0576 | 0.7689 | 0.3278 | 0.1919 |
| `mid_update_arc_split_d20_split15_rot30` | 0.7333 | 0.3112 | 1.0859 | 0.8456 | 0.1671 |

结论：

- 立即更新最差，说明把新轨迹直接写进正在 denoise 或即将 denoise 的区域，会产生明显 OOD。
- 延迟更新有效，`d20` 比 `d0` 明显降低 `ADE`、`arc_ADE` 和 `chamfer`。
- 但即便 `d20`，`post` 仍然偏高，说明“延迟更新”只能缓解条件突变，不能根治中途转向能力不足。
- `arc_split_d20` 的 `chamfer` 低于 `d0/d5`，说明路径形状更贴近，但时间相位 / 速度 / 朝向仍然不稳。

工程判断：

- web_demo 如果要稳定，应该考虑把手动 `update_traj` 延迟到 `commit + horizon` 或至少 `commit + chunk_size` 后生效。
- 这会带来交互延迟，但能避免当前正在去噪的 latent 被旧轨迹和新轨迹连续拉扯。
- 彻底解决需要训练阶段加入 mid-session update 分布，而不是只靠推理补丁。

## 7. web_demo 侧已经明确的问题

### 7.1 clear trajectory

问题：

- 清空轨迹后，如果 `_traj_buf` 不 reset，旧轨迹 buffer 会继续影响后续 step。
- 即使 `traj_emb=None`，如果 CFG denoise 仍然调用 ControlNet，也不是纯 backbone。

当前修正方向：

- `web_demo/model_manager.py` 在 `traj_input is None` 时 reset `_traj_buf`。
- `models/diffusion_forcing_wan.py` 的 `_denoise_with_cfg()` 增加 `traj_emb is None` 的 backbone-only 分支。

残余风险：

- 已经被旧轨迹影响过的 latent 不会回滚，所以 clear 后仍可能有短期惯性。
- 需要重启 web_demo 才能确保加载最新代码。

### 7.2 手动 update_traj 的空间对齐

问题：

- 用户画的新轨迹是世界坐标。
- 如果不平移到当前 root 附近，`TrajStreamBuffer` 做 anchor-subtract 后，模型看到的可能是从原点跳到远处的轨迹。

当前修正方向：

- 手动非 timestamped 轨迹进入 `update_trajectory()` 时，如果首点离当前 root 有偏移，则 prepend 当前 root。
- repeat / debug 轨迹也应该走同一套“以当前 root 为 anchor 重新放置”的逻辑。

### 7.3 默认轨迹时间

当前倾向：

- web_demo 用户画的是空间路径，不要求手动给时间戳。
- 默认把用户画的路径按弧长重采样成 100 个点，对应 5 秒。
- 等价于 `waypoint_dt = 0.05s`，20 FPS。
- 如果 plan 超时，不应该简单 clamp 到最后一个点；需要 repeat 或末端速度外推，否则模型会停在终点。

## 8. 当前总判断

已经确认：

- `stream_generate_step()` 在 GT-root 条件下可以正常工作。
- 轨迹条件是有效的，无轨迹明显变差。
- history length 不是主要瓶颈。
- pred-root 闭环是 web_demo 长时间推理的主要误差来源。
- 手动 update 的最大风险是：
  - 轨迹空间 anchor 没有对齐当前 root。
  - 新轨迹过早写入正在 denoise 的区域。
  - 模型训练时没有见过 mid-session 轨迹突变。

短期工程策略：

- web_demo 默认使用 `0.05s` waypoint spacing。
- 用户画线后按弧长均匀采样。
- 新轨迹整体平移到当前 root。
- update 生效时间不要立即写入，优先测试 `commit + horizon`。
- clear 轨迹时必须 reset buffer 并走 backbone-only。

中期训练策略：

- 需要 finetune 一个“流式轨迹更新”分布：
  - 随机 split token。
  - 保留历史 window。
  - 在未来 suffix 上做旋转 / 替换 / 速度变化。
  - self-forcing rollout 中用 pred-root 更新 anchor。
  - 让模型见到带噪 anchor 和中途目标变化。

后续优先级：

1. 先把 web_demo 的手动 update 逻辑稳定到与 diagnose 的最佳设置一致。
2. 再用 BABEL 长程样本做多文本、多轨迹切换视频验证。
3. 最后再决定是否启动专门的 stream-update finetune。
